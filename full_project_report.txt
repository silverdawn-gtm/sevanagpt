1.1  WHAT IS SEVANAGPT?
────────────────────────

SevanaGPT is an AI-powered Indian government scheme discovery platform that
helps citizens find, browse, and understand 4,668 government welfare schemes
across 22 Indian languages. The platform combines:

  • Hybrid search (keyword + semantic via pgvector)
  • FSM-based conversational chatbot with LLM integration
  • Multilingual translation via IndicTrans2 (GPU-accelerated) with
    Google Translate fallback
  • Eligibility matching engine that scores user profiles against schemes
  • Voice input/output via Web Speech API

The name "Sevana" (സേവന) means "service" in Malayalam, reflecting the
project's goal of making government services accessible to all Indians
in their native language.

4.4  SERVICES LAYER
───────────────────

SERVICE: chat_service.py
  Purpose: Orchestrates the conversational chatbot
  Core Function: process_message(session, session_id, user_message, language) → dict
  Flow:
    1. Load or create Conversation record
    2. Extract accumulated context (age, gender, state, category, occupation)
    3. Classify user intent via LLM (7 intents: greeting, search_scheme,
       ask_detail, check_eligibility, clarify, goodbye, other)
    4. Determine next FSM state based on intent + current state
    5. Generate LLM response with state-specific prompt
    6. Search schemes if in SCHEME_SEARCH state
    7. Return: reply, schemes[], suggestions[], fsm_state
  Multi-Language:
    - Pre-defined suggestions in 12 languages
    - User message translated to English for LLM processing
    - Response generated in target language via LLM instruction
  Supporting Functions:
    - get_chat_history(session, session_id) → message list
    - reset_conversation(session, session_id) → deletes conversation

SERVICE: search_service.py
  Purpose: Hybrid search combining keyword and semantic search
  Functions:
    keyword_search(session, query, limit=20)
      - Splits query into terms (min 3 chars each)
      - ILIKE search on name, description, benefits, eligibility_criteria
      - Score = matching_terms / total_terms
    keyword_search_translated(session, query, lang, limit=20)
      - Searches scheme_translations for non-English queries
    semantic_search(session, query, limit=20)
      - Embeds query via Mistral (1024-dim)
      - Cosine distance search on scheme_embeddings
      - Score = 1.0 - distance
    reciprocal_rank_fusion(*result_lists, k=60)
      - RRF formula: score = 1/(k + rank + 1) per list
      - Deduplicates by scheme ID, sums scores
    hybrid_search(session, query, filters..., limit=10)
      - Calls keyword_search + semantic_search
      - Merges via RRF
      - Applies post-filters (category, level, state, ministry)
      - Returns top 10

SERVICE: embedding_service.py
  Purpose: Generate and manage Mistral embeddings
  Functions:
    scheme_to_text(scheme) → str
      - Concatenates: name + description + "Benefits: " + benefits +
        "Eligibility: " + eligibility_criteria
      - Truncates to 8000 chars
    generate_embeddings_batch(session, schemes, batch_size=10) → int
      - Batch of 10 schemes per API call
      - Checks text_hash to skip unchanged schemes
      - Exponential backoff on 429: 5s, 10s, 20s, 40s, 80s (5 attempts)
      - Deletes stale rows before inserting new
      - 1s sleep between batches for rate limiting

SERVICE: mistral_service.py
  Purpose: LLM provider management (Mistral primary, Groq fallback)
  Provider Detection:
    - mistral_configured() → bool (checks MISTRAL_API_KEY)
    - groq_configured() → bool (checks GROQ_API_KEY)
    - chat_configured() → bool (either provider)
  Embedding Functions:
    - embed_texts(texts) → list[list[float]] (Mistral only)
    - embed_single(text) → list[float]
  Chat Functions:
    - _groq_chat(messages, temperature, max_tokens) → str
      OpenAI-compatible API via httpx, 30s timeout
    - _mistral_chat(messages, temperature, max_tokens) → str
      Mistral SDK
    - chat_complete(messages, temperature=0.7, max_tokens=1024) → str
      Tries Mistral first, falls back to Groq
  Intent Classification:
    - classify_intent(user_message, context) → {"intent": str, "entities": dict}
      7 intents: greeting, search_scheme, ask_detail, check_eligibility,
      clarify, goodbye, other
      temperature=0.1, max_tokens=256

SERVICE: translate_service.py
  Purpose: On-demand translation with aggressive caching
  Supported Languages: 22 (11 original + 11 IndicTrans2 additional)
  Engine Priority: IndicTrans2 → Google Translate → return original
  Key Functions:
    translate_text(text, tgt_lang, db) → str
      - Cache check (SHA-256 hash key)
      - IndicTrans2 first, Google Translate fallback
      - 8s timeout per translation
      - Caches result in translation_cache
    translate_texts_batch(texts, tgt_lang, db) → list[str]
      - Batch cache lookup
      - Chunks by 4500 char limit
      - Newline concatenation for batch Google Translate (~13x faster)
      - 20s overall timeout
      - Caches all new translations
  Google Translate Special Codes:
    mni → mni-Mtei (Manipuri with script subtag)
    kok → gom (Goan Konkani)
    bodo, sat → NOT SUPPORTED (IndicTrans2 only)

SERVICE: eligibility_service.py
  Purpose: Match user profiles against scheme eligibility criteria
  Functions:
    check_eligibility(session, profile) → list[dict]
      - Loads all active schemes
      - Filters by state if provided
      - Computes match_score (0-1) based on matching criteria count
      - Returns results sorted by score desc
    _compute_match(scheme, profile) → (float, list[str])
      - Checks: gender, age range, social category, income ceiling,
        disability, student, BPL status
      - Returns: (matched/total, matched_criteria descriptions)
    get_eligibility_options(session) → dict
      - Returns available: genders, social_categories, states

SERVICE: indictrans_client.py
  Purpose: HTTP client for IndicTrans2 microservice
  Design: Returns None on all failures (triggers fallback gracefully)
  Functions:
    is_available() → bool (GET /health)
    translate_single(text, target_lang) → str | None
      POST /translate, timeout: INDICTRANS_TIMEOUT
    translate_batch(texts, target_lang) → list[str] | None
      POST /translate/batch, timeout: INDICTRANS_BATCH_TIMEOUT


4.5  CHATBOT FSM (FINITE STATE MACHINE)
────────────────────────────────────────

File: backend/app/chatbot/fsm.py

  States:
    GREETING         Welcome user, show topic suggestions
    NEED_EXTRACTION  Ask clarifying questions (age, state, category)
    SCHEME_SEARCH    Present matching schemes
    DISAMBIGUATION   Handle ambiguous queries
    SCHEME_DETAIL    Show full scheme details
    CLOSING          End conversation

  Intents (classified by LLM):
    greeting         User says hello
    search_scheme    User wants to find schemes
    ask_detail       User wants scheme details
    check_eligibility User wants eligibility check
    clarify          User provides additional info
    goodbye          User ends conversation
    other            Anything else

  Transition Map:
    FROM              INTENT              → TO
    ────────────────  ──────────────────  ─────────────────
    GREETING          greeting            → GREETING
    GREETING          search_scheme       → NEED_EXTRACTION
    GREETING          check_eligibility   → NEED_EXTRACTION
    GREETING          other               → NEED_EXTRACTION
    NEED_EXTRACTION   search_scheme       → SCHEME_SEARCH
    NEED_EXTRACTION   check_eligibility   → SCHEME_SEARCH
    NEED_EXTRACTION   other               → SCHEME_SEARCH
    SCHEME_SEARCH     ask_detail          → SCHEME_DETAIL
    SCHEME_SEARCH     clarify             → DISAMBIGUATION
    SCHEME_SEARCH     other               → DISAMBIGUATION
    DISAMBIGUATION    search_scheme       → SCHEME_SEARCH
    DISAMBIGUATION    clarify             → SCHEME_SEARCH
    DISAMBIGUATION    other               → DISAMBIGUATION
    SCHEME_DETAIL     search_scheme       → NEED_EXTRACTION
    SCHEME_DETAIL     other               → NEED_EXTRACTION
    CLOSING           greeting            → GREETING
    CLOSING           search_scheme       → NEED_EXTRACTION
    ANY               goodbye             → CLOSING

File: backend/app/chatbot/prompts.py

  System Prompt:
    "SevanaGPT: helpful government scheme assistant for India"
    Guidelines: friendly, clear, ask clarifying questions, structured info

  Per-State Prompts:
    greeting_prompt(language)
      - Welcome + topic suggestions (education, agriculture, housing, etc.)
    need_extraction_prompt(context, language)
      - Include accumulated context, ask for missing info
    scheme_search_prompt(schemes_context, user_query, context, language)
      - Show top 3-5 schemes with name, brief description, key benefit

      - Full scheme information display

7.3  CACHING STRATEGY
─────────────────────

  Table: translation_cache
  Key: SHA-256 hash of "{text}|{src_lang}|{tgt_lang}"
  Fields: source_text, translated_text, src_lang, tgt_lang, created_at

  Cache Flow:
    1. On every translation request: check cache by hash_key
    2. On cache hit: return immediately (zero latency)
    3. On cache miss: translate via engine chain, then cache result
    4. Batch cache lookup: IN query on multiple hash_keys (efficient)

  Pre-Population:
    - Startup: Background pre-translation of Hindi scheme names
    - CLI: python -m app.data.run_translations --all (all 22 languages)
    - MyScheme API: 11 languages of scheme_translations from API


7.4  BATCH OPTIMIZATION
───────────────────────

  Google Translate Batching:
    - Concatenates texts with newline separator
    - Single API call translates all texts at once
    - Splits response by newline to recover individual translations
    - ~13x faster than individual calls
    - Fallback: individual translation on split mismatch
    - Chunk limit: 4500 characters per batch

  IndicTrans2 Batching:
    - GPU batch processing (MAX_BATCH_SIZE=4)
    - Padding to longest sequence in batch
    - OOM fallback to one-by-one

  Backend Batch Processing:
    - Batch cache lookup (IN query)
    - Parallel workers (CONCURRENT_WORKERS=2 in pre_translate)
    - Periodic commits to avoid memory buildup
    - Overall timeout: 20s for batch operations

================================================================================
  CHAPTER 9 — SEARCH SYSTEM
================================================================================

9.1  HYBRID SEARCH ARCHITECTURE
────────────────────────────────

  The search system combines two complementary strategies:

    User Query
      │
      ├──→ [Keyword Search]  → Ranked list 1 (term matching)
      │
      ├──→ [Semantic Search] → Ranked list 2 (vector similarity)
      │
      └──→ [RRF Merger]      → Final ranked results
                                  │
                                  ▼
                          [Post-Filters] → Top 10 results
                          (category, state, ministry, level)


9.2  KEYWORD SEARCH
───────────────────

  Algorithm:
    1. Split query into terms (min 3 chars each)
    2. For each term, ILIKE search on:
       - scheme.name
       - scheme.description
       - scheme.benefits
       - scheme.eligibility_criteria
    3. Score = matching_terms / total_terms
    4. Return top 20 sorted by score descending

  Translated Search:
    For non-English queries, additionally searches scheme_translations table
    on translated name and description fields.


9.3  SEMANTIC SEARCH (pgvector)
───────────────────────────────

  Algorithm:
    1. Embed query text using Mistral embed API (1024-dim vector)
    2. Cosine distance search against scheme_embeddings table
    3. Score = 1.0 - cosine_distance
    4. Return top 20 nearest neighbors

  Embedding Model: mistral-embed (1024 dimensions)
  Storage: pgvector Vector(1024) column type


9.4  RECIPROCAL RANK FUSION (RRF)
──────────────────────────────────

  Formula: score(d) = Σ 1/(k + rank_i(d) + 1)
    Where k=60 (tuning parameter), rank_i(d) = position of document d in list i

  Properties:
    - Schemes appearing in BOTH keyword and semantic lists get boosted
    - Higher-ranked items in each list contribute more
    - k=60 balances between top-heavy and flat scoring
    - Deduplicates by scheme ID, sums scores from all lists

  Example:
    Scheme X at rank 1 in keyword + rank 3 in semantic:
    Score = 1/(60+1+1) + 1/(60+3+1) = 0.0161 + 0.0156 = 0.0317

    Scheme Y at rank 1 in keyword only:
    Score = 1/(60+1+1) = 0.0161

    → Scheme X ranks higher (appears in both lists)


9.5  EMBEDDING GENERATION
─────────────────────────

  Process:
    1. For each scheme, build text:
       name + description + "Benefits: " + benefits + "Eligibility: " + eligibility
    2. Truncate to 8000 chars
    3. Compute SHA-256 hash of text
    4. Check if existing embedding has same hash (skip if unchanged)
    5. Call Mistral embed API in batches of 10
    6. Store 1024-dim vector + hash in scheme_embeddings table

  Rate Limit Handling:
    - Exponential backoff: 5s, 10s, 20s, 40s, 80s (5 attempts)
    - 1s delay between batches
    - Stale row cleanup before insert

  Statistics:
    Total schemes:     4,668
    Embedding dim:     1,024
    Batch size:        10
    Est. generation:   ~8-10 minutes (with rate limits)

10.2  KNOWN ISSUES & GOTCHAS
─────────────────────────────

  1. IndicTrans2 + transformers: Must pin transformers<4.45.0 — newer versions
     break the custom tokenizer (_special_tokens_map error).

  2. use_cache=False required: Custom IndicTrans2 model has past_key_values bug
     with newer transformers. Must set use_cache=False in generate().

  3. peft version: Must pin peft>=0.7.0,<0.11.0 for LoRA compatibility.

  4. Docker BuildKit networking: huggingface_hub may report misleading
     "connection" errors when real issue is 403 (token permissions).

  5. Mistral rate limits: Free tier hits 429 frequently. Embedding generation
     uses exponential backoff (5s, 10s, 20s, 40s, 80s).

  6. GPU OOM: Reduce CONCURRENT_WORKERS (default 2) and BATCH_SIZE (default 8)
     in pre_translate.py. For indictrans, reduce MAX_BATCH_SIZE (default 4).

  7. MyScheme API duplicates: Returns duplicate slugs — deduplicates by
     scheme_id in ingest_translations.py.

  8. docker compose up --build kills exec: Never rebuild while docker exec
     tasks are running. Use docker cp to hot-patch.

  9. Google Translate special codes: mni→mni-Mtei, kok→gom. bodo and sat
     have NO Google Translate support.

  10. HF_TOKEN hardcoded: Currently committed in docker-compose.yml as
      plaintext. Should be externalized to env var.

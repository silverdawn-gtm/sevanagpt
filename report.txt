1. INTRODUCTION
--------------------------------------------------------------------------------

SevanaGPT is an AI-powered web application designed to help Indian citizens
discover, understand, and check their eligibility for government welfare schemes.
India has thousands of government schemes spread across central and state
governments, managed by over 50 ministries, targeting diverse demographics.
Citizens often struggle to find schemes they are eligible for due to language
barriers, information fragmentation, and complex eligibility criteria.

SevanaGPT addresses this by providing:
  - A conversational AI assistant that understands user needs in 23 Indian
    languages and recommends relevant schemes.
  - An eligibility-matching engine that scores schemes against user profiles.
  - Hybrid search (keyword + semantic) for scheme discovery.
  - Voice input/output for accessibility in low-literacy populations.
  - A comprehensive browsable catalog of 4,700+ schemes.

The platform aggregates data from official government sources including
MyScheme.gov.in, data.gov.in, Kaggle datasets, and HuggingFace Hub.


2. OBJECTIVES
--------------------------------------------------------------------------------

  a) Build an intelligent, multilingual scheme discovery platform that serves
     citizens across all Indian states and union territories.

  b) Implement conversational AI that can gather user context (age, gender,
     location, income, occupation) through natural dialogue and recommend
     matching schemes.

  c) Support 23 Indian languages with real-time translation using both
     cloud APIs and a locally-hosted neural machine translation model.

  d) Provide voice-based interaction (speech-to-text and text-to-speech) for
     accessibility.

  e) Aggregate scheme data from multiple government data sources into a unified,
     searchable database with structured eligibility fields.

  f) Implement hybrid search combining keyword matching and vector-based
     semantic similarity for accurate scheme discovery.

5. METHODOLOGY
--------------------------------------------------------------------------------

5.1 Data Collection and Ingestion

  The platform aggregates scheme data from multiple authoritative sources through
  an automated ingestion pipeline:

  Step 1: Seed Reference Data
    - 18 scheme categories (Agriculture, Education, Health, etc.)
    - 36 states and union territories with ISO codes
    - 100+ central and state-level ministries
    - 30+ cross-cutting tags

  Step 2: MyScheme.gov.in API
    - Official Indian government scheme portal
    - Batch fetching: 50 schemes per request with 300ms delay
    - Yields ~4,600 schemes with eligibility details
    - Auto-maps to categories and states

  Step 3: HuggingFace/Embedded Dataset
    - Pre-curated dataset of central government schemes
    - Supplements official data with additional descriptions

  Step 4: Kaggle & data.gov.in
    - Public datasets for broader coverage
    - Credential-based authentication
    - Duplicate detection via slug matching

  Step 5: Embedding Generation
    - Mistral-embed model generates 1024-dimensional vectors
    - Text input: scheme name + description + benefits + eligibility
    - Change detection via SHA256 hash (re-embeds only when content changes)
    - Stored in pgvector for similarity search

  Step 6: Pre-Translation
    - Background task on server startup
    - Pre-translates scheme names for Hindi (most requested language)
    - Populates translation cache for instant access

5.2 Conversational AI (Finite State Machine)

  The chat system uses a Finite State Machine (FSM) with 6 states to manage
  structured conversations:

  +----------+     +-----------+     +---------------+
  | GREETING +---->+ NEED      +---->+ SCHEME        |
  |          |     | EXTRACTION|     | SEARCH        |
  +----------+     +-----------+     +-------+-------+
                                             |
                   +----------------+  +-----+--------+
                   | CLOSING        |<-+ SCHEME       |
                   |                |  | DETAIL       |
                   +----------------+  +--------------+
                          ^
                   +------+-------+
                   | DISAMBIG-    |
                   | UATION       |
                   +--------------+

  State Transitions:
  - GREETING: Welcome message, detect initial intent
  - NEED_EXTRACTION: Gather user profile (age, gender, state, category,
    occupation, income) through natural dialogue
  - SCHEME_SEARCH: Search database using accumulated context, present results
  - DISAMBIGUATION: Clarify ambiguous requests
  - SCHEME_DETAIL: Show detailed information for a selected scheme
  - CLOSING: End conversation

  Intent Classification:
  - Primary: LLM-based classification via Mistral/Groq
  - Fallback: Keyword matching in 12 languages for intents: greeting,
    search_scheme, ask_detail, check_eligibility, clarify, goodbye

  Entity Extraction:
  - LLM extracts structured fields from natural language:
    age, gender, state, category, occupation, income
  - Context accumulates across conversation turns
  - Used for filtering scheme recommendations

5.3 Hybrid Search (Keyword + Semantic + RRF)

  The search system combines two retrieval strategies using Reciprocal Rank
  Fusion (RRF):

  Query ----+---> Keyword Search (ILIKE pattern matching)
            |         |
            |         v
            |     Ranked List A
            |         |
            +---> Semantic Search (pgvector cosine similarity)
                      |
                      v
                  Ranked List B
                      |
            +---------+---------+
            |    RRF Merge      |
            |  score = SUM(     |
            |   1/(k + rank))   |
            |  k = 60           |
            +-------------------+
                      |
                      v
                Final Results (deduplicated, filtered)

  Keyword Search:
  - Splits query into terms (minimum 3 characters)
  - ILIKE matching across name, description, benefits, eligibility fields
  - Scores by matching term frequency

  Semantic Search:
  - Generates query embedding via Mistral-embed (1024 dimensions)
  - Cosine distance similarity against scheme embeddings in pgvector
  - Returns top-k most semantically similar schemes

  RRF Merge:
  - Combines both ranked lists using formula: score = SUM(1/(60 + rank))
  - Deduplicates by scheme ID
  - Applies post-filters (category, state, ministry, level)

5.4 Eligibility Matching Engine

  A structured matching algorithm scores schemes against user profiles:

  Criteria (7 fields):
  1. Gender match (exact match or scheme accepts "all")
  2. Age range (min_age <= user_age <= max_age)
  3. Social category (SC, ST, OBC, General, EWS)
  4. Income threshold (user_income <= scheme_max_income)
  5. Disability status
  6. Student status
  7. BPL (Below Poverty Line) status

  Scoring:
  - match_score = matched_criteria / total_applicable_criteria
  - Range: 0.0 to 1.0 (higher = better match)
  - Results sorted descending by match score
  - Each result includes list of matched criteria for transparency

5.5 Multi-Tier Translation System

  Translation uses a cascading strategy for performance and reliability:

  Request ---> [1] Pre-computed Cache (scheme_translations table)
                  |  (instant, zero cost)
                  | miss
                  v
              [2] Runtime Cache (translation_cache table, SHA256 hash key)
                  |  (instant, zero cost)
                  | miss
                  v
              [3] IndicTrans2 Microservice (self-hosted GPU)
                  |  (fast, no external dependency)
                  | fail/timeout
                  v
              [4] Google Translate via deep-translator (free fallback)
                  |  (reliable, rate-limited)
                  v
              Cache result in translation_cache table

  Batch Optimization:
  - Multiple texts concatenated with newline separators
  - Single API call translates all at once (~13x speedup)
  - Falls back to individual translation if batch split fails
  - Concurrent chunk processing for large batches (4500 char limit)

5.6 Voice Pipeline

  Full voice interaction is supported via the Bhashini API:

  User Audio ---> ASR (Speech-to-Text)
                    |
                    v
                  User text (in native language)
                    |
                    v
                  Translate to English (NMT)
                    |
                    v
                  Chat AI processing (in English)
                    |
                    v
                  Translate reply to user's language (NMT)
                    |
                    v
                  TTS (Text-to-Speech)
                    |
                    v
                  Audio response to user

7. API ENDPOINTS
--------------------------------------------------------------------------------

Base URL: /api/v1

  Scheme Discovery:
  - GET  /schemes                    List schemes (paginated, filterable)
  - GET  /schemes/featured           Top 12 featured schemes
  - GET  /schemes/{slug}             Scheme detail with FAQs
  - GET  /categories                 List all 18 categories
  - GET  /categories/{slug}          Category schemes (paginated)
  - GET  /states                     List all 36 states/UTs
  - GET  /states/{slug}              State schemes (paginated, level filter)
  - GET  /ministries                 List all ministries
  - GET  /ministries/{slug}          Ministry schemes (paginated)

  Search:
  - POST /search                     Hybrid keyword + semantic search
  - GET  /search/suggest             Autocomplete suggestions

  Chat:
  - POST /chat/message               Send message, get AI response + schemes
  - GET  /chat/history/{session_id}  Retrieve conversation history
  - POST /chat/reset/{session_id}    Reset conversation state

  Voice:
  - POST /chat/voice                 Full voice pipeline (ASR+NMT+Chat+TTS)
  - POST /voice/transcribe           Speech-to-text
  - POST /voice/synthesize           Text-to-speech

  Eligibility:
  - POST /eligibility/check          Match user profile against schemes
  - GET  /eligibility/options        Get form options (genders, categories)

  Translation:
  - POST /translate                  Translate text between languages
  - GET  /languages                  List supported languages

  Health:
  - GET  /health                     Service health check

  All endpoints accept a `lang` parameter (ISO 639-1 code) to return
  translated content. Default is English.

11. KEY DESIGN DECISIONS
--------------------------------------------------------------------------------

  1. Async-First Architecture: FastAPI + asyncpg enables high concurrency
     for handling multiple simultaneous translation and chat requests.

  2. Self-Hosted Translation: IndicTrans2 microservice provides Indian
     language translation without external API dependencies or costs,
     optimized for Indic scripts.

  3. Hybrid Search over Pure Semantic: Combining keyword + semantic search
     with RRF ensures both precision (exact term matches) and recall
     (semantically similar schemes).

  4. FSM for Chat: Finite State Machine provides predictable, structured
     conversations rather than open-ended LLM responses, ensuring users
     receive actionable scheme recommendations.

  5. Multi-Tier Translation Cache: Aggressive caching (pre-computed +
     runtime hash cache) minimizes translation latency and API costs.

  6. pgvector over External Vector DB: Using PostgreSQL's pgvector extension
     avoids the complexity of a separate vector database while providing
     efficient similarity search.

  7. Graceful Degradation: Every external service (LLM, translation, voice)
     has a fallback path, ensuring the platform remains functional even when
     individual services are unavailable.

  8. Microservice Isolation: IndicTrans2 runs as a separate service, allowing
     independent scaling, GPU allocation, and deployment.

  9. Mobile-First Design: Responsive UI ensures accessibility on smartphones,
     the primary internet device for most Indian citizens.

  10. Data Source Aggregation: Combining MyScheme.gov.in, Kaggle, data.gov.in,
      and HuggingFace ensures comprehensive scheme coverage.


12. CHALLENGES AND SOLUTIONS
--------------------------------------------------------------------------------

  Challenge: Model Size and GPU Memory
  Solution: Used the distilled 200M parameter variant of IndicTrans2 which
  fits within 4GB VRAM (GTX 1650) using float16 precision.

  Challenge: Translation Latency for Real-Time UI
  Solution: Multi-tier caching (pre-computed, runtime hash cache), newline
  batching (~13x speedup), and concurrent chunk processing.

  Challenge: Diverse Indian Scripts and Languages
  Solution: IndicTransToolkit handles script-specific preprocessing and
  postprocessing; BCP-47 Flores language codes for accuracy.

  Challenge: Structured Eligibility Data
  Solution: Schemes store structured fields (target_gender, age range, income
  threshold, social category) enabling SQL-based matching without LLM costs.

  Challenge: Conversation Context Management
  Solution: FSM with JSONB context field accumulates user profile across
  multiple conversation turns, persisted in database.

  Challenge: Search Quality
  Solution: RRF merge of keyword and semantic search provides better ranking
  than either method alone.

15. CONCLUSION
--------------------------------------------------------------------------------

SevanaGPT demonstrates how modern AI technologies -- large language models,
neural machine translation, vector embeddings, and speech processing -- can be
combined to solve a real civic problem: connecting Indian citizens with the
government welfare schemes they are entitled to.

The platform successfully aggregates 4,700+ schemes from official sources,
supports 23 Indian languages through a self-hosted neural translation model
(IndicTrans2), and provides multiple discovery paths: conversational AI chat,
eligibility matching, semantic search, and traditional browsing. The FSM-based
chat system ensures structured, goal-oriented conversations that gather user
context and deliver actionable scheme recommendations.

The architecture prioritizes accessibility (voice support, 23 languages,
mobile-first design), reliability (graceful degradation with fallback chains),
and performance (multi-tier caching, batch optimization, async processing).
By hosting the translation model locally on consumer GPU hardware (GTX 1650),
the platform minimizes external API dependencies and operational costs while
maintaining real-time translation capabilities.

SevanaGPT serves as a practical example of applying AI for social good,
bridging the information gap between government welfare programs and the
citizens who need them most.
